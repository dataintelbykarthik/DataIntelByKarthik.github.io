Snowflake Patent Solves the Stale Statistics Problem

Query optimizers rely on cached statistics from the last ANALYZE, creating a "danger window" of stale data — a nightly ETL loads 50M rows at 2 AM, but by 6 AM the optimizer still assumes yesterday's row counts and picks a broadcast join that now moves 300GB unnecessarily. The patent's HJD (Hybrid Join Decision) operator fixes this by measuring actual row counts at runtime instead of relying on stale cached statistics.

How Probe-Side Annotations Work

The optimizer defers strategy selection until runtime by embedding three components into the query plan: a threshold calculated from warehouse capacity and memory constraints, a watch list monitoring runtime signals like bloom filter performance, and a decision table mapping outcomes to strategies. Query plans become "conditional programs" rather than fixed artifacts — the probe side waits for the build side to report actual row counts before the Join Decision Manager picks broadcast or hash-join.

The Three-Operator Architecture: HJB, HJD, HJP

Snowflake restructured its join operator by splitting one monolithic operator into three single-responsibility components: HJB (Hash-Join-Build) constructs the hash table and reports actual row counts; HJD (Hash-Join-Decision) takes those cardinality signals and decides broadcast vs. local sync; HJP (Hash-Join-Probe) executes the join using whichever method HJD selected. The patent's real value is the operator boundary itself — it's what makes deferring the decision to runtime possible.
